# CSS Audit & Refactoring Plan — Windrose Standalone Conversion

**Created:** 2026-05-24
**Scope:** Comprehensive CSS modernization + namespace unification (dmt- → windrose-)
**Browser Target:** Tier 2 (Modern baseline: last 2 versions / >0.5% usage)
**Skill:** css-modern-features (rushenn/css-modern-features)

---

## Overview

This plan consolidates five phases of CSS refactoring:
- **Phase 0:** Resolve settingsStyles.css duplication
- **Phase 1:** Rename CSS custom properties (--dmt- → --windrose-)
- **Phase 2:** Rename CSS class names (dmt-* → windrose-*) across 63 files
- **Phase 3:** Rename CustomEvent names (dmt-* → windrose-*)

Total refactoring scope: **1 CSS file + 63 JS/TS files, ~1,200+ line changes**

## Phase 4: CSS Modernization (Tier 2)

**Scope:** styles.css only (1 file)
**Browser Target:** Tier 2 (modern baseline)
**Testing:** Manual visual testing (light + dark themes), then E2E suite

### Phase 4a: Color & Theming

**Objectives:**
- Replace hardcoded hex colors with OKLCH color space
- Use `light-dark()` for automatic dark mode handling
- Use `color-mix()` for derived colors (hover, active states)
- Reduce color duplication

**Current State (in styles.css):**
- 2 hardcoded hex colors in custom properties
- 78+ hardcoded hex colors throughout the file
- 176+ hardcoded rgba values
- Zero `light-dark()` or `color-mix()` usage
- Zero OKLCH color space usage
- Zero theme detection (@media prefers-color-scheme)

**Patterns to Replace:**

#### Pattern 1: Hardcoded background colors
```css
/* Before */
background: #ffffff;
@media (prefers-color-scheme: dark) {
  background: #1a1a1a;
}

/* After */
background: light-dark(oklch(0.98 0 0), oklch(0.15 0 0));
```

#### Pattern 2: Hover/active states
```css
/* Before */
.dmt-button { background: #4a9eff; }
.dmt-button:hover { background: #3a8eef; }

/* After */
.windrose-button { background: oklch(0.6 0.15 250); }
.windrose-button:hover { background: color-mix(in oklch, currentColor 20%, oklch(0.6 0.15 250)); }
```

#### Pattern 3: Opacity variations
```css
/* Before */
background: rgba(196, 165, 123, 0.3);

/* After */
background: oklch(from var(--windrose-border-primary) l c h / 0.3);
```

**Specific Changes:**

| Location | Current | New | Reason |
|----------|---------|-----|--------|
| Line 71 | `--windrose-border-primary: #c4a57b` | `--windrose-border-primary: oklch(0.68 0.12 35)` | Convert to OKLCH |
| Line 74 | `--windrose-border-secondary: #3a3a3a` | `--windrose-border-secondary: oklch(0.22 0 0)` | Convert to OKLCH |
| Line 79 | `--windrose-accent-blue: #4a9eff` | `--windrose-accent-blue: oklch(0.6 0.15 250)` | Convert to OKLCH |
| Line 80 | `--windrose-accent-blue-hover: #3a8eef` | Use `color-mix()` instead | Derive from base |
| Line 81 | `--windrose-accent-red: #8b0000` | `--windrose-accent-red: oklch(0.35 0.15 10)` | Convert to OKLCH |
| Line 82 | `--windrose-accent-red-hover: #a00000` | Use `color-mix()` instead | Derive from base |
| All hex colors | Various hardcoded colors | OKLCH + `light-dark()` | Systematic replacement |

**Color Conversion Strategy:**

For each hardcoded hex color:
1. Identify the color's semantic role (background, border, text, accent)
2. Convert hex to OKLCH using color converter
3. If color has light/dark variants, use `light-dark()` function
4. If color has hover/active variants, use `color-mix()` instead of separate variables

**Files to Change:**
- `styles.css` — All color declarations

**Testing Protocol:**
1. Build: `npm run build`
2. Deploy: `npm run deploy`
3. Manual test in Obsidian:
   - Open Windrose plugin
   - Verify colors in light theme
   - Switch to dark theme, verify colors
   - Hover over interactive elements, verify hover states
   - Check contrast ratios (text is readable)
4. If colors look wrong, adjust OKLCH values and re-test

**Risk:**
**Medium** — Color changes are visible and subjective. Dark mode must be tested. OKLCH conversions require verification.

### Phase 4b: Layout & Sizing

**Objectives:**
- Replace fixed widths with fluid sizing using `clamp()`, `min()`, `max()`
- Replace hardcoded breakpoints (768px) with container queries
- Improve mobile responsiveness

**Current State:**
- Rigid pixel-based layouts throughout
- Fixed padding/margin values
- @media query at 768px breakpoint
- No `clamp()`, `min()`, `max()` usage
- No container queries

**Patterns to Replace:**

#### Pattern 1: Responsive widths
```css
/* Before */
width: 100%;
max-width: 400px;
margin: 0 auto;

/* After */
width: min(100%, 400px);
```

#### Pattern 2: Responsive padding
```css
/* Before */
padding: 24px;
@media (max-width: 768px) {
  padding: 16px;
}

/* After */
padding: clamp(1rem, 3cqw, 1.5rem);
```

#### Pattern 3: Dynamic sizing
```css
/* Before */
width: 100%;
height: 400px;

/* After */
width: 100%;
height: clamp(300px, 50vh, 500px);
```

**Specific Changes:**

Search styles.css for patterns:
- `max-width: \d+px` → Replace with `min(100%, <value>px)`
- `width: 100%; max-width:` → Replace with `width: min(100%, <value>px)`
- `padding: \d+px` → Consider `clamp()` for responsive padding
- `@media (max-width: 768px)` → Evaluate if container queries fit better

**Files to Change:**
- `styles.css` — Layout and sizing rules

**Testing Protocol:**
1. Build and deploy
2. Resize browser window and test responsive behavior
3. Test on mobile viewport (375px to 1920px)
4. Verify that layouts remain readable at all sizes

**Risk:**
**Low-Medium** — Layout changes are usually safe, but test at multiple viewport sizes.

### Phase 4c: Selectors & Logic

**Objectives:**
- Use CSS nesting to reduce selector repetition
- Use `:has()`, `:is()`, `:where()` for more flexible targeting
- Introduce `@layer` for cascade clarity
- Improve selector specificity management

**Current State:**
- Verbose descendant selectors (5+ levels deep)
- No CSS nesting
- One existing `:has()` usage
- No `:is()`, `:where()`, or `@layer`

**Patterns to Replace:**

#### Pattern 1: CSS Nesting
```css
/* Before */
.windrose-modal { /* ... */ }
.windrose-modal .windrose-modal-header { /* ... */ }
.windrose-modal .windrose-modal-header .windrose-modal-title { /* ... */ }

/* After */
.windrose-modal {
  /* ... */
  .windrose-modal-header {
    /* ... */
    .windrose-modal-title {
      /* ... */
    }
  }
}
```

#### Pattern 2: :is() for grouped selectors
```css
/* Before */
.windrose-button,
.windrose-icon-button,
.windrose-tool-button {
  color: var(--windrose-text-primary);
}

/* After */
:is(.windrose-button, .windrose-icon-button, .windrose-tool-button) {
  color: var(--windrose-text-primary);
}
```

#### Pattern 3: :has() for conditional styling
```css
/* Before */
.windrose-item { margin-bottom: 8px; }
.windrose-item:last-child { margin-bottom: 0; }

/* After */
.windrose-item:has(+ .windrose-item) { margin-bottom: 8px; }
```

#### Pattern 4: @layer for cascade control
```css
@layer windrose-base, windrose-components, windrose-utilities;

@layer windrose-base {
  /* Base element styles */
}

@layer windrose-components {
  /* Component styles */
}
```

**Specific Changes:**

1. **Identify nesting candidates** — Look for selectors that are 3+ levels deep
2. **Convert to nested format** — Reduce nesting levels
3. **Extract common selectors** — Use `:is()` for grouped rules
4. **Introduce @layer** — Clarify CSS cascade (optional, low priority)

**Files to Change:**
- `styles.css` — Selector patterns

**Testing Protocol:**
1. Visual inspection — No functional changes, output should be identical
2. Build and deploy
3. Manual testing in Obsidian — Verify all components look the same
4. E2E tests should pass without changes

**Risk:**
**Low** — Refactoring selectors has no functional impact. CSS output should be identical.

- Phase 0 is quick and unblocks everything

- Phase 2 is complex and benefits from being last (other changes are already tested)
