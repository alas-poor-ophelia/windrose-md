**Planning Document**

## Overview

Version 1.5.x is the "Dungeon" release: refining and completing the dungeon grid mapping functionality. The goal is to ship 4-6 minor point releases (1.5.1, 1.5.2, etc.) with focused, achievable features before moving on to other areas.

**Target scope per release**: 1-3 day features, with one potential week-long feature.

## Feature Specifications

### 1. Deep Linking (Copy Link to Location)

**Priority**: High
**Scope**: 1-2 days
**Release**: 1.5.1
**Status**: ✅ **COMPLETE**

#### Problem
DMs write session notes in separate Obsidian files and need to reference specific map locations. Currently no way to link directly to a position on a map.

#### Solution
Add "Copy link" action for objects, labels, and arbitrary positions. Uses custom markdown link syntax that renders cleanly in notes.

#### Link Format
```
[Location name](windrose:mapId,x,y,zoom,layerId)
```
Example: `[Throne Room](windrose:castle-level-1,15,8,1.2,layer_001)`

#### Click Behavior
1. If map note not open → open it first
2. Switch to specified layer
3. Center view on coordinates at 115-120% zoom
4. (Target location is centered, no additional highlight needed)

#### Copy Link UI
- **Selection toolbar**: Add "Copy link" button when object/label selected
- Copies link with selected item's name as link text

#### Implementation
- Use `registerMarkdownPostProcessor()` to intercept `windrose:` links in rendered markdown
- Transform clicks into navigation calls that open note + pan/zoom the map view
- Layer is always included in link (captures specific view of specific thing on specific layer)

#### Technical Notes
- Custom link processor provides clean in-Obsidian UX
- Links are readable in source mode (not opaque URIs)
- Could add `obsidian://` URI support later for external app integration

#### Files Likely Affected
- `src/main.ts` - Register markdown post processor
- `src/components/SelectionToolbar.tsx` - Add copy link button
- `src/hooks/useMapData.ts` - Handle navigation from link clicks
- `src/utils/deepLinkHandler.ts` - New file for link generation/parsing
- `src/hooks/useEventCoordinator.ts` - Emit/receive navigation events

#### Future Extension
- Object-to-object links (click object → navigate to linked object)
- Inter-map links (open different map file)
- External app support via `obsidian://` URI scheme

### 2. Layer Transparency

**Priority**: High
**Scope**: 1-2 days
**Release**: 1.5.1
**Status**: ✅ **COMPLETE** (shipped with Deep Linking)

#### Problem
Multi-floor dungeons are supported via layers, but there's no way to see how floors align. Users must mentally track where stairs lead.

#### Solution
Toggle to show the layer directly below the current layer at user-adjustable opacity.

#### Implementation
- Add `showLayerBelow: boolean` toggle in layer context menu (or layer controls header)
- **Opacity slider** (not fixed percentage) - default 25%, range 10-50%
- When enabled, render the layer immediately below at selected opacity
- Render before current layer (underneath)
- Only show cells/edges, not objects/labels (to avoid clutter)
- Visual indicator in layer list when transparency is active

#### Files Likely Affected
- `src/components/LayerControls.tsx` - Toggle UI + opacity slider
- `src/hooks/useCanvasRenderer.ts` - Additional render pass
- `src/types/MapData.ts` - Add `showLayerBelow` and `layerBelowOpacity` to UI preferences

#### Future Extension
- Show multiple layers below
- Per-layer opacity controls

### 3. Flood Fill Tool

**Priority**: Medium-High
**Scope**: 1-2 days
**Release**: 1.5.3
**Decision**: ✅ Confirmed

#### Problem
Drawing a room requires painting each cell individually or using rectangle/circle tools. Organic room shapes require tedious cell-by-cell work.

#### Solution
Click inside an enclosed area to fill all connected empty cells.

#### Implementation
- New tool in shape tools section
- Click on empty cell → flood fill algorithm
- Respects current color/opacity
- Boundaries: filled cells, edges of visible area
- Consider: maximum fill limit to prevent accidents (e.g., 500 cells)
- Undo support (single action)

#### Algorithm
Standard 4-directional flood fill on cell grid. Check neighbors via geometry abstraction (works for both grid and hex).

#### Files Likely Affected
- `src/components/ToolPalette.tsx` - Flood fill tool button
- `src/hooks/useShapeDrawing.ts` - Flood fill logic
- `src/utils/cellAccessor.ts` - Cell neighbor queries

### 4. Quick Door Placement Mode

**Priority**: Medium
**Scope**: 2-3 days
**Release**: 1.5.4
**Decision**: ✅ Confirmed

#### Problem
Doors are the most common dungeon object. Current workflow: select object tool → scroll sidebar to find door → click → drag to position on edge. Too many steps for something so frequent.

#### Solution
Dedicated door placement mode where clicking on edges places doors with correct rotation.

#### Implementation
- New tool or object tool sub-mode
- Click on edge between cells → place door object
- Auto-rotate to match edge orientation
- Modifier key (Shift) → place secret door instead
- Visual preview on hover showing door placement
- Works on edges between filled and empty cells

#### Files Likely Affected
- `src/components/ToolPalette.tsx` - Door tool
- `src/hooks/useObjectInteractions.ts` - Edge-click detection
- `src/utils/edgeOperations.ts` - Edge position/rotation calculation

#### Considerations
- Edge detection needs to identify the edge from click position
- Doors should snap to edge centers
- Support both vertical and horizontal edges

### 5. Enhanced Dungeon Generator

**Priority**: High
**Scope**: ~2.5 days
**Release**: 1.5.2
**Decision**: ✅ Confirmed - All three features

#### Problem
Generator creates rectangular rooms and straight corridors. Doesn't showcase the segment API for diagonal walls. No environmental variety (water, etc.). Generated dungeons are empty—no objects.

#### Solution
Three enhancements (priority order):
1. **Diagonal corridor support using segments** (primary goal)
2. **Water features in generated dungeons** (secondary)
3. **Random object placement** (treasure, traps, furniture)

#### Implementation: Diagonal Corridors (~1 day)

- Modify corridor carving to allow 45-degree paths
- Path-find at sub-cell level (considering segments)
- Paint corridor segments instead of full cells
- Requires careful border calculation at transitions

This showcases the existing segment infrastructure in a dramatic way and is the primary goal of this release.

#### Implementation: Water Features (~0.5 day)

- Add `waterChance` parameter to generator config
- Some rooms randomly designated as "water rooms"
- Water cells painted in blue with reduced opacity
- Styles can influence water: Cavern (underground lakes), Crypt (flooded crypts)
- Water rendered as regular cells with specific color

New style possibilities:
- **Sewer**: High water chance, narrow corridors, grates
- **Cavern**: Underground pools, organic shapes
- **Flooded**: Entire lower areas under water

#### Implementation: Random Object Placement (~1 day)

- Add object placement pass after room/corridor generation
- Configurable: object density slider or "sparse/moderate/dense" presets
- Object types: treasure chests, traps, barrels, crates, tables, etc.
- Placement rules: avoid doorways, prefer room corners/centers
- Respect room size (small rooms get fewer objects)
- Style-specific objects (e.g., sarcophagi in Crypt, mushrooms in Cavern)

#### Files Likely Affected
- `src/features/generator/dungeonGenerator.js` - Core algorithm changes
- `src/features/generator/corridorCarver.js` - Diagonal pathing
- `src/features/generator/objectPlacer.js` - New file for object placement
- `src/features/generator/styles.js` - New style definitions + style-specific objects
- `src/components/GeneratorModal.tsx` - New options UI

## Design Philosophy

### "From Map to Adventure"

Current Windrose excels at *drawing* dungeons. The 1.5.x features focus on the bridge to *running* them:

- **Deep linking** = Reference locations from session notes
- **Layer transparency** = Navigate multi-floor during play
- **Quick doors** = Reduce prep friction
- **Flood fill** = Speed up room creation

Generator enhancements serve *creation*, but the other features serve *play*—which is where the dungeon ultimately lives.

### Principles

1. **Leverage existing infrastructure**: Segments, layers, objects are mature. Build on them.
2. **DM workflow first**: Features should speed up session prep or table play.
3. **Ship incrementally**: Each release should feel complete, not half-baked.
4. **Avoid scope creep**: Save "full visions" for 2.0.

---

## Deferred to Future Releases

These ideas are interesting but too large for 1.5.x:

- **Room numbering**: Auto-increment room labels - not compelling enough for 1.5.x
- **Freehand polygons**: Requires rethinking cell model
- **Manual curve editing**: Complex UI, unclear value
- **Object-to-object linking**: Good, but deep linking is higher priority
- **Copy/paste regions**: Useful but not dungeon-essential
- **Secret passage rendering**: Could be 1.5.x if quick, otherwise defer
